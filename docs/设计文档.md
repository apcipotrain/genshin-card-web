## 二、路由设计

### 2.1 页面流转图

```
Login ──→ Home ──→ Chapters ──→ Game (PVE)
              │                    │
              │                    └──→ Result
              │
              └──→ Match ──→ Waiting ──→ Game (PVP)
                                 │       │
                                 │       └──→ Result
                                 └──→ Match (返回)
```

### 2.2 Router 实现

```typescript
export type PageId = 'home' | 'chapters' | 'match' | 'waiting' | 'game' | 'result' | 'login';

class Router {
  navigate(page: PageId, params?: Record<string, unknown>): void
  back(): void
  getState(): RouteState
  onNavigate(fn: (state: RouteState) => void): () => void
}
```

- 无 URL Hash，纯内存状态管理
- 支持历史回退（单层）
- 支持路由参数传递

### 2.3 页面生命周期

```typescript
interface Page {
  render(): HTMLElement;       // 创建DOM（仅一次）
  show(): void;                 // 显示页面
  hide(): void;                 // 隐藏页面
  onEnter?(state: RouteState): void;  // 路由进入回调
}
```

## 三、UI 设计

### 3.1 设计语言

- **主色调**：深蓝黑 `#0f0c29` → `#1a1a2e`（原神深渊风格）
- **强调色**：金色 `#ffd700`（七天神像/原石色）
- **功能色**：
  - 绿色 `#4caf50` — PVE / 成功 / 存活
  - 红色 `#f44336` — PVP / 伤害 / 阵亡
  - 蓝色 `#4a9eff` — 信息 / AI
  - 橙色 `#ff9800` — 警告 / 可交互

### 3.2 游戏界面 8 人布局（逆时针排列）

```
        [top[2]] [top[1]] [top[0]]     ← 上方3人 (逆时针远→近)
  [left[0]]                     [right[1]]   ← 左右各2人
  [left[1]]                     [right[0]]
                    [self]                    ← 自己 (右下)
```

**座位顺序说明**：以自己（index hIdx）为起点（右下），逆时针一圈：
- right 列：从下到上 `hIdx+1`, `hIdx+2`
- top 行：从右到左 `hIdx+3`, `hIdx+4`, `hIdx+5`（渲染时反转为 `+5, +4, +3`）
- left 列：从上到下 `hIdx+6`, `hIdx+7`

回合顺序与视觉逆时针一致（数组索引递增）。

### 3.3 PVE 座位布局

PVE 使用 `pveSeatIndex` 标记座位，`getPVESeatLayout(totalPlayers)` 动态分配：

| 人数 | 有效座位索引 | 空位 |
|------|-------------|------|
| 8人 | 0,1,2,3,4,5,6,7 | 无 |
| 7人 | 0,1,2,3,5,6,7 | 对角(4) |
| 6人 | 0,1,2,3,5,7 | 4,6 |
| 5人 | 0,1,3,5,7 | 2,4,6 |
| 4人 | 0,3,5,7 | 1,2,4,6 |
| 3人 | 0,3,5 | 1,2,4,6,7 |
| 2人 | 0,4 | 其余 |

## 四、数据流设计

### 4.1 PVE 游戏状态管理

```
GamePage (客户端本地)
    │
    ├── EventBus（事件总线）──→ UI 更新
    ├── DeckManager（牌堆）
    ├── Player[]（玩家状态，含 faction: Ally/Enemy）
    ├── CardEffectManager（卡牌效果）
    ├── EquipEffectManager（装备效果）
    ├── DamageSystem（伤害系统）
    ├── SkillManager（武将技能）
    └── IPlayerDriver（玩家驱动）
            ├── HumanWebUIDriver（人类玩家，Promise 交互）
            └── DelayedAIDriver（AI 玩家，可配置延迟）
```

### 4.2 PVP 游戏状态管理

```
服务端 GameHost
    │
    ├── GameFlowController（主控）
    │   ├── EventBus → registerEventForwarding → Socket.IO → 客户端
    │   ├── DeckManager / CardEffectManager / DamageSystem ...
    │   └── IPlayerDriver
    │       ├── DelayedAIDriver（AI 玩家，1200ms 延迟）
    │       └── RemotePlayerDriver（真人玩家，Socket 通信）
    │
    └── 客户端收到 game_event → processPVPSingleEvent → 渲染

客户端 GamePage
    ├── SocketManager（Socket.IO 客户端单例）
    ├── handlePVPPrompt → HumanWebUIDriver（复用 PVE 的 UI 交互）
    └── processPVPSingleEvent → 渲染战场/动画
```

### 4.3 UI 渲染模式

- **PVE 事件驱动渲染**：UI 通过 `EventBus` 监听游戏事件，实时响应状态变化
- **PVP 事件驱动渲染**：服务端 `EventBus.on('*')` 全量转发到客户端，客户端 `processPVPSingleEvent` 处理
- **人类玩家交互**：`HumanWebUIDriver` 通过 Promise 等待玩家操作；PVP 模式通过 `setPVPRespond()` 注入 socket 回调，PVE/PVP 共用同一套 UI 方法
- **AI 玩家**：`DelayedAIDriver` 包装 `AIDriver`，PVE 支持 0.3s-5.0s 可调延迟，PVP 固定 1200ms

## 六、PVP 联机架构

### 6.1 架构

```
Browser (Client) ←→ WebSocket ←→ GameServer (Node.js)
                                      │
                                      ├── AccountManager（账号+经验）
                                      ├── RoomManager（房间管理）
                                      ├── GameHost（游戏实例）
                                      │   ├── GameFlowController
                                      │   └── RemotePlayerDriver / DelayedAIDriver
                                      └── Socket.IO（事件转发）
```

### 6.2 通信协议

- WebSocket 长连接（Socket.IO）
- JSON 消息格式
- 事件类型：`login` / `signup` / `auth_token` / `create_room` / `join_room` / `toggle_ready` / `start_hero_select` / `select_monarch_hero` / `select_hero` / `prompt` / `prompt_response` / `game_start` / `game_event` / `game_over` / `add_exp`

### 6.3 身份脱敏

- 服务端 `sanitizePlayerForViewer` 对每个真人玩家独立脱敏
- 非主公、非自己的玩家 `role` 隐藏为 `'Unknown'`
- 游戏结束后 `game_over` 事件包含所有玩家真实身份 (`playerRoles`)

### 6.4 断线重连

- 断线时 `GameHost.handleDisconnect` 将真人玩家替换为 AI 接替
- 标记 `escapedPlayerIds`，结算时逃跑玩家经验为 0
- 重连时恢复 `RemotePlayerDriver`，清除逃跑标记

## 七、经验与等级系统

### 7.1 等级表

| 等级范围 | 升级所需经验 |
|----------|-------------|
| Lv.1-55 | 当前等级 × 100 |
| Lv.56 | 56,000 |
| Lv.57 | 57,000 |
| Lv.58 | 58,000 |
| Lv.59 | 59,000 |
| Lv.60 | 封顶 |

### 7.2 经验值含义

- `exp`：累计经验总量（从 Lv.1 开始累计）
- `level`：当前等级
- `currentLevelExp`：当前等级内已有经验（= exp - 该等级累计阈值）
- `nextLevelExp`：升下一级所需总经验（该等级的升级需求）

### 7.3 经验来源

- **PVP 结算**：`GameHost.computeExpForAllPlayers()` 计算所有真人玩家经验
  - `baseExp`：胜利/失败基础经验（按角色和胜利方）
  - `bonusExp`：击杀奖励（按击杀目标身份计算）
  - `totalExp`：`baseExp + bonusExp`（本局获得量）
  - 逃跑者：经验为 0
- **PVE 结算**：客户端 `computeLocalPVPExp` 计算本地经验，通过 `add_exp` 事件上报服务器

### 7.4 客户端计算

`SocketManager.setAccount()` 收到账号数据时，自动从 `exp` 计算 `currentLevelExp` 和 `nextLevelExp`（与服务端 `AccountManager.getLevelAndProgress` 逻辑一致）。
